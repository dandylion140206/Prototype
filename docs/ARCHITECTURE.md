# Architecture

## 目的

このドキュメントは、Godot 4.xプロジェクトにおける設計判断の基準を定義する。

特定のScene、Node、Script、Resource、クラスの一覧を管理することは目的としない。

現在の具体的な実装についてはコードとSceneを正とし、このドキュメントでは、新しい機能を追加または既存機能を変更するときの設計原則を扱う。

---

# 1. 基本原則

Godotの設計では、エンジンが提供する仕組みを優先して利用する。

主な構成要素は以下とする。

* Node
* Scene
* SceneTree
* Signal
* Resource
* RefCounted
* Callable
* Autoload

一般的なオブジェクト指向設計やSOLID原則は補助的な判断基準として利用する。

ただし、一般的な設計パターンをGodotへそのまま持ち込むことを目的にせず、Node、Scene、Signal、ResourceなどGodot固有の仕組みに自然に沿う設計を優先する。

設計原則そのものより、以下を重視する。

* 責務が理解しやすいこと
* 依存関係が追跡できること
* 変更の影響範囲を限定できること
* Sceneを再利用しやすいこと
* 不要な抽象化がないこと

---

# 2. Scene

## 2.1 Sceneは可能な限り自己完結させる

再利用可能なSceneは、可能な限り自身が必要とする構成を内部に持つ。

Sceneが使用される外部環境の具体的なNode構造を前提にしない。

推奨:

```text
Parent / Owner
    └── ReusableScene
            ├── InternalNode
            └── InternalNode
```

再利用可能なScene自身が外部SceneTreeを探索して必要なNodeを探す構成は避ける。

```text
ReusableScene
    ↓
外部SceneTreeを探索
    ↓
特定のNodeを取得
```

外部依存が必要な場合は、その関係を把握している上位コンテキストから提供する。

---

## 2.2 Scene内部を実装詳細として扱う

Scene内部のNode構造は、そのSceneの実装詳細として扱う。

Scene外部から内部の深いNodeを直接操作する構造を可能な限り避ける。

推奨:

```text
External
    ↓
Scene Root
    ↓
Internal Nodes
```

Scene外部へ公開する必要がある操作は、Scene Rootなど明確な公開境界を通して提供する。

これにより、内部Node構造の変更が外部へ波及することを抑える。

---

## 2.3 Sceneを分離する基準

以下の場合はSceneとして分離することを検討する。

* 独立して生成または破棄される
* 複数箇所で再利用される
* 独自のライフサイクルを持つ
* 単独で動作確認したい
* 内部構造を外部から隠したい
* 意味のあるNodeのまとまりとして扱える

Node数やScript行数だけを理由にSceneを分割しない。

---

# 3. SceneTree

SceneTreeは画面上の位置関係だけではなく、所有関係とライフサイクルの関係として考える。

あるNodeが親Nodeの存在に依存し、親とともに生成・破棄されることが自然であれば、親子関係として配置することを検討する。

親の存在に依存しないNodeを、管理上の都合だけで不必要に深い子Nodeとして配置しない。

SceneTreeの構造そのものを、複数システム間の暗黙的なAPIとして利用しない。

---

# 4. Node

## 4.1 Nodeを使用する基準

以下のようなSceneTree上の機能が必要な場合はNodeを利用する。

* SceneTreeのライフサイクル
* `_enter_tree()`
* `_ready()`
* `_process()`
* `_physics_process()`
* Transform階層
* Node間のSignal
* Sceneとしての構成

SceneTree上に存在する必要がないデータや処理まで、すべてNodeとして実装しない。

---

## 4.2 Nodeの責務

NodeおよびScriptには、可能な範囲で焦点の定まった責務を持たせる。

1つのNodeが多くの独立した責務を持つようになった場合は、分離を検討する。

分離先の候補:

* 子Node
* 独立Scene
* Resource
* RefCounted
* その他の専用クラス

ただし、小さく密接に関連する処理まで機械的に分割しない。

単一責任原則は補助的な判断基準として利用し、分割自体を目的にしない。

---

# 5. 所有関係

複数オブジェクトの関係を管理する必要がある場合は、それらを所有または把握しているオブジェクトが関係の構築を担当することを基本とする。

例:

```text
Owner
├── ChildA
└── ChildB
```

`ChildA`と`ChildB`の連携が必要な場合:

```text
ChildA
    ↓ Signal
Owner
    ↓ Method
ChildB
```

のように、両者を知っている`Owner`が仲介または接続を担当することを検討する。

兄弟Node同士がSceneTreeを探索して直接参照を取得する構成を常態化させない。

---

# 6. Dependency Injection

SceneやNodeが外部オブジェクトを必要とする場合は、依存先を自身で探索するより、外部から明示的に提供する方法を優先する。

利用できる方法には以下がある。

* `@export`による参照
* 公開プロパティ
* 初期化メソッド
* Signal接続
* Callable
* Resource
* 生成時の設定

依存関係を提供する責務は、その関係を把握している所有者または上位コンテキストが持つ。

ただし、Scene内部で完結できる依存関係まで外部注入しない。

自己完結できるSceneは自己完結させることを優先する。

---

# 7. Signal

Signalはイベントを通知するために利用する。

発信側は、可能な限り受信側の具体的な実装を知る必要がない状態を保つ。

```text
Emitter
    ↓ Signal
Listener
```

Signalは、オブジェクト間の直接参照を減らし、疎結合なイベント通知を実現するために利用する。

---

## 7.1 Signalとメソッド

概ね以下の基準で使い分ける。

イベントの通知:

```text
何かが起きた
```

にはSignalを検討する。

処理の要求:

```text
何かを実行する
```

には通常のメソッドを検討する。

処理そのものを外部から提供する必要がある場合はCallableを検討する。

Signalを単なる間接的なメソッド呼び出しとして乱用しない。

---

## 7.2 Signal接続

Signalの発信側と受信側を両方把握している所有者が存在する場合は、その所有者が接続を担当することを検討する。

```text
Owner
├── Emitter
└── Receiver
```

```text
Emitter.signal
    ↓
Ownerが接続
    ↓
Receiver.method
```

これによりEmitterとReceiverの直接依存を減らす。

---

# 8. Callable

Callableは、処理を外部から提供または差し替える必要がある場合に利用を検討する。

Signalがイベント通知に適しているのに対し、Callableは「どの処理を実行するか」を外部から提供する用途に適する。

単純な所有関係では通常のメソッド呼び出しを優先し、不要な間接化を避ける。

---

# 9. Resource

Resourceは主にデータを表現するために利用する。

利用を検討する例:

* 設定データ
* 再利用可能なデータ定義
* Inspectorで編集するデータ
* 複数のSceneから利用するデータ
* 保存や読み込みの対象となるデータ

NodeとしてSceneTree上に存在する必要がないデータを、機械的にNodeとして実装しない。

---

## 9.1 Resourceとロジック

Resourceにロジックを持たせること自体は禁止しない。

自身が保持するデータに関する処理など、SceneTreeへの依存が不要なロジックはResourceに持たせることができる。

SceneTreeの状態や特定Nodeのライフサイクルへ強く依存する処理は、Node側へ配置することを検討する。

---

## 9.2 共有Resource

Resourceは複数の利用者から同じインスタンスを参照される可能性がある。

共有Resourceを実行時に変更する場合は、その変更が他の参照元へ影響する可能性を考慮する。

以下を区別する。

* 共有する定義データ
* 個別インスタンスが持つ実行時状態

共有設定と個別状態を不用意に混在させない。

---

# 10. RefCounted

SceneTree上のライフサイクルやTransformを必要とせず、Resourceとして保存する必要もない軽量オブジェクトにはRefCountedの利用を検討する。

選択の目安:

```text
Node
    SceneTree上で動作する必要がある

Resource
    データとして保存・共有・Inspector編集したい

RefCounted
    SceneTree外の軽量な実行時オブジェクト
```

用途に対して最も単純な選択肢を利用する。

---

# 11. Autoload

Autoloadは、現在のSceneに依存せず常に存在する必要があるシステムや、Scene間で維持する必要があるデータに利用する。

利用を検討する例:

* Scene間で維持する状態
* Scene遷移を管理する仕組み
* プロジェクト全体で必要な設定
* 永続的なシステム

Autoloadを任意のNode参照を保存するためだけのグローバル変数置き場として使用しない。

通常のSceneまたはNodeの所有関係で解決できる場合は、その方法を優先する。

---

# 12. UI

UIは主に以下を担当する。

* 表示
* ユーザー入力の受付
* UI固有の状態

ゲームロジックの中心をUIへ配置しない。

基本的な情報の流れは以下を検討する。

```text
Game Logic
    ↓ State / Signal
UI
```

ユーザー操作:

```text
UI
    ↓ Signal / Method
Controller or Owner
    ↓
Game Logic
```

UIがSceneTree全体を探索してゲーム状態を取得する構成を可能な限り避ける。

---

# 13. 継承とComposition

継承とCompositionのどちらを使用するかは、責務と関係性に基づいて判断する。

GodotではNodeやSceneの組み合わせによる構成が自然な場合が多いため、機能を組み合わせる場合はCompositionを検討する。

継承は、明確な型の関係があり、派生クラスを基底クラスとして扱っても契約が破綻しない場合に利用する。

以下の状態になった場合は設計を見直す。

* 基底クラスが多数の責務を持つ
* 派生クラスごとの条件分岐が基底クラスへ増える
* 一部の派生クラスでは基底機能を使用できない
* 継承階層が深くなり、挙動を追跡しにくい

SOLID原則はこの判断を補助するために利用するが、過度な抽象化は避ける。

---

# 14. プロジェクト構成

Godotは固定されたプロジェクトディレクトリ構造を要求しない。

関連するScene、Script、Resource、Assetを保守しやすい単位でまとめる。

新しいファイルの配置は、以下を考慮して判断する。

* どのSceneまたは機能に属するか
* 他の場所から再利用されるか
* 既存プロジェクトの構成と一貫しているか

ファイル名やフォルダ名だけでアーキテクチャ上の責務を決定しない。

サードパーティー製のアドオンやプラグインについては、Godotの一般的な構成との整合性を考慮する。

---

# 15. システム間通信

システム間通信を設計するときは、まず所有関係を確認する。

判断順序の目安:

```text
1. 同じオブジェクト内部で完結するか
        ↓
2. 親子の明確な所有関係があるか
        ↓
3. 通常のメソッド呼び出しで十分か
        ↓
4. イベント通知としてSignalが適切か
        ↓
5. 処理の注入としてCallableが適切か
        ↓
6. Sceneを超える永続システムが本当に必要か
        ↓
7. 必要な場合のみAutoload等を検討
```

最初からグローバルな通信機構を導入しない。

---

# 16. 設計判断の流れ

新しい機能を追加するときは、概ね以下の順序で検討する。

```text
既存実装で実現できるか
        ↓
責務を誰が所有するか
        ↓
SceneTree上に存在する必要があるか
        ↓
Node / Scene / Resource / RefCountedを選択
        ↓
外部依存が必要か
        ↓
必要なら所有者から依存を提供
        ↓
通信方法を選択
Method / Signal / Callable
        ↓
必要最小限の実装
```

新しいManager、Singleton、基底クラス、抽象化レイヤーを最初の選択肢にしない。

---

# 17. ドキュメント方針

アーキテクチャドキュメントでは、「現在何が存在するか」ではなく「どのように設計判断するか」を記録する。

以下は原則としてコードとSceneを正とする。

* Scene一覧
* Node一覧
* Script一覧
* クラス一覧
* メソッド一覧
* プロパティ一覧

以下は必要に応じてドキュメント化する。

* システムの責務
* システム境界
* システム間の契約
* 重要な依存方向
* コードだけでは分からない制約
* 設計を採用した理由
* 意図的なトレードオフ

個別システムが複雑になった場合は、必要に応じて専用ドキュメントを追加する。

```text
docs/
├── ARCHITECTURE.md
├── CODE_STYLE.md
└── systems/
    └── <system_name>.md
```

個別システムが追加されるたびに、このドキュメントへ具体的なクラスやSceneを追記しない。

---

# 18. 最終原則

設計では以下を優先する。

```text
明確な責務
    ↓
自己完結したScene
    ↓
自然な所有関係
    ↓
明示的な依存関係
    ↓
適切な通信方法
    ↓
必要最小限の抽象化
```

設計パターンやSOLID原則を適用すること自体を目的にしない。

Godotの仕組みに自然に沿い、コードとSceneを確認することで責務と依存関係を理解でき、変更による影響範囲を限定できる設計を優先する。
